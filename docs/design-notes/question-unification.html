A. The principle — and why the current state inverts it

One survey structure renders the questions; the map is a read-only projection of that same structure; and every accessible question is a first-class modular question on the correct branch, executed through the single mutate() write seam.

The intent is stated in the manifest header itself: “map == runtime by construction” — both the runtime and the dashboard read the same manifest array, so editing one file moves both steps/manifest.ts:1–6. That guarantee currently holds only at the inter-phase manifest level. It does not hold for the survey-flow section of the map, and it does not describe where the mature experiences actually live.

The inversion. The unification refactor pulled the less-mature paths into the registry + map (the per-phase *.modular.yaml question batteries) while the mature, canonical experiences — the build-list character editor, the prefill confirmation, the track chooser, and the carve / mechanism / touch galleries — stayed as hand-built React components and were left out of the unified data structure entirely. The map therefore advertises the paths users mostly don’t take and is silent about the paths they do.

Three concrete symptoms confirm the inversion against current code:

  1. The map never reads the manifest. The survey-flow section is built only from the four per-phase modular YAMLs — FLOW_SOURCES DashboardView.tsx:48–54 — via buildModularFlowGraph(). It never calls buildManifestStepGraph() buildStepGraph.ts:237; that function is exercised only by tests / completeness.ts. So the editor-step galleries (carve, mechanisms, touch in steps/manifest.ts:69–119) get no node, and the “stub (gallery / wizard step)” legend swatch DashboardView.tsx:114 is dead — nothing ever emits that kind.
  2. The default flow isn’t in any data source. Phase B has two parallel implementations in PhaseB.tsx: the hand-built BuildListView (the default, PhaseB.tsx:690, 535–629) and the “Step by step” manual variant that runs the pb_* registry modules from phase_b_characters.modular.yaml (PhaseB.tsx:712–740). The map shows only the manual variant’s 55 pb_* nodes — i.e. the path most users never see.
  3. Data and runtime are aligned in exactly one phase. The full non-identity Phase A (PhaseA.tsx / phase_a_identity.modular.yaml) is shown in the map but never rendered: StudioShell.tsx:18 imports IdentityLite, Prefill, PhaseB, PhaseF — not PhaseA. The app runs only the identity_lite head. Phase F is the only phase where the data structure and the runtime align.

There is also vestigial machinery already in place for the right answer: computeReserveNodes() buildStepGraph.ts:150–182 emits library-not-in-flow nodes (the “reserve” badge) for registry modules absent from the live flow — but it is currently empty, because every registered module is listed in its YAML. The concept (a library of inert-but-preserved questions) exists; it just isn’t being used to hold the genuinely-reserve batteries.

C. The map as a projection — “can’t drift by construction”

Today the survey-flow section reads four YAML files (FLOW_SOURCES DashboardView.tsx:48–54) that are parallel to the runtime’s ordering source (manifest). Two ordering sources ⇒ drift is possible. The fix is to make the flow section a projection of the same structures the runtime renders, at two levels:

  1. Inter-phase level (the spine): render the flow section from buildManifestStepGraph() buildStepGraph.ts:237, which already produces exactly one node per manifest entry (steps/manifest.ts:69) with spine / fork / join / data edges. This is the function the dashboard does not yet call. Adding the carve / mechanisms / touch editor-steps to the map is then automatic — they are already in the manifest.
  2. Intra-phase level (drill-down): for each question-step phase, render the per-phase question graph (buildModularFlowGraph) as a drill-down under its manifest node, keyed off the same registry the SurveyRunner resolves through (questionRegistry questions/registry.ts). The discovery-method gate (§B, B-build-list vs B-step-by-step) becomes a real branch node, so the default path is the one shown.

What “can’t drift by construction” means concretely

  • Single ordering source. The map’s node set for the spine is manifest.map(...); the runtime advances through the same array (nextSpineStepAfter StudioShell.tsx:319–344). There is no second list to forget to update. This already satisfies C8 (“node set == runtime step set; zero ghost/missing”).
  • Single question source. Drill-down nodes resolve through questionRegistry — the exact map the SurveyRunner uses. A new question lands in questions/<phase>/<id>.ts + its sub-registry + the phase YAML order; it then appears in both the runtime and the map with no map edit. The fan-out rule is already documented in questions/registry.ts.
  • No hand-maintained diagram. Remove the FLOW_SOURCES array as the primary source; the modular graphs remain only as drill-downs derived from the registry. The dead “stub (gallery / wizard step)” legend DashboardView.tsx:114 becomes live (editor-steps now emit nodes), and computeReserveNodes buildStepGraph.ts:150 finally has content (the library batteries from §E) to render with the existing “reserve” badge.
  • Enforced by the completeness checks already shipped. C1–C7 (completeness.ts) run over the same manifest graph; C3 rejoin, C5 orphan-inputs, and C7 unreachable will fail visibly in the dashboard if a promoted question declares an input no upstream step writes, or a branch dead-ends. Drift becomes a red banner, not a silent divergence.
Net: “adding a question automatically appears on the map” reduces to “the map projects manifest + questionRegistry, and so does the runtime.” The projection is read-only and derives 100% from the structures the runtime executes.

D. mutate() gap analysis

The mutate() seam (flag VITE_KM_MUTATE_SEAM, default off flags/mutateFlag.ts) is a single pure-patch write path: a module returns Partial<KeyboardIR>, the reducer applies it as a path-scoped deep merge restricted to the module’s declared writes, with fail-fast whole-patch rejection on any undeclared path (M1–M6, mutateApply.ts:189–213; contract contracts/mutate-seam.contract.md). What it supports today:

  • Single-value, single-step writes. One answer → one patch → merge. The wired writers are the 5 identity/header modules (header.bcp47 / header.name / header.copyright: primary_script, iso_code, language_name_english, pa_copyright_holder, plus the il_* lite variants) and one Phase B writer, pb_standard_letters → stores[] questions/b/pb_standard_letters.ts:105–125.
  • Idempotent re-answer (M4) and read snapshot via MutateContext.ir survey/types.ts:115–120.
  • Touch re-propagation off the staleness closure on physical-lock completion reducer.ts:228–243, and touch IR + per-key provenance first-class in KeyboardIR.touchLayout (US2–US5, #825).

What the mature flows need that mutate() doesn’t yet do:

It currently uses dedicated store mutators (deleteNode/restoreNode/deleteItem/restoreItem/restoreAll/keepAll) CarveGallery.tsx:28–72 that maintain a deletion overlay + undo stack. mutate() returns a value patch and the reducer’s setWorkingIR is explicitly overlay-preserving StudioShell.tsx:462–468 — but no carve module declares writes or produces such a patch yet. Contract / version implications

  • If a new IR location is added (a carve-overlay field, or a confirmedInventory field on KeyboardIR), that touches the Engine-owned locked packages/contracts surface ⇒ a contracts bump + §18 joint engine+content sign-off, exactly as the touch-provenance field required in #822/#825 (the pre-1.0 convention maps a spec-MAJOR contract change to a minor bump; 0.13.0 → 0.14.0). plan.md:11–16, 66–71
  • If existing IR locations suffice (carve expressed over existing layers[].rows[].keys[]; inventory expressed over stores[] like pb_standard_letters), no contract change is needed — only new modules with declared writes + mutate(). This is the preferred path; pursue it first.
  • No mutate() signature change is needed for batching: a single mutate() can return a patch touching many array indices, all authorized by one writes: [layers[].rows[].keys[]] declaration. The seam is already powerful enough for multi-leaf writes; the gap is declaring the surfaces and routing the galleries through the reducer instead of the direct store mutators.

D2. Looping / iterative question-series — a gap the unification must close

New requirement (Matt). The mature character-placement / build-list flows may need to be decomposed into separate and possibly looping question series — e.g. an iterative branch “for each character, ask this sub-series” (per-character or per-key), not just a flat linear set. Flattening these into a linear question list would lose the looping behaviour. The modular question model does not support iteration today, so this is a first-class part of migrating the character flows.

(1) Does the current schema / FlowGotoRule / registry support loops or repeated sub-flows?

No — and the runtime actively forbids them. Verified against current code:

  • The question schema has no iteration construct. FlowQuestion.next is string | null | FlowGotoRule[] survey/types.ts:53; FlowGotoRule is {condition?, goto, default?} survey/types.ts:30–34. There is no loop-entry/exit marker, no per-iteration index, no “repeat while”, and no binding of a question to a collection. FlowDef survey/types.ts:61–67 is a flat questions[] + an optional provenance_questions[] — no sub-flow / nested-flow type exists.
  • The runner treats revisiting a node as an error, not a loop. advanceThrough() keeps a Set<string> visited guard whose sole purpose is to break out of cycles: “a visited guard prevents an infinite loop if a YAML author creates a cycle (A→B→A)” SurveyRunner.tsx:180–205. A back-edge logs "cycle detected" and terminates the flow. So even the “loop back” goto options that exist today (e.g. pb_additional_methods offering “loop back for text sample / linguist / picker” questions/b/pb_additional_methods.ts:38–40) are routing back to an earlier question id, not a bounded iteration with per-pass state — and they rely on the visited-guard not tripping.
  • Completeness treats cycles as a hard error. C2 (findCycles) flags any cycle in the writes→inputs data graph as a hard error completeness.ts:182–185, 216–229. A naive “loop a question back to itself” would both terminate the runner and turn the dashboard banner red.
  • Collections are captured flat, not iterated. The build-list pattern gathers a whole character set in one multi_select/char-list answer (confirmedInventory, PhaseB.tsx:55–71, 607–612) — there is no “for each grapheme, run a sub-series” mechanism anywhere in the survey layer.

(2) What a looping / iterative question-series primitive would need to look like

A first-class loop construct on the modular model (additive to FlowQuestion/FlowDef) would need:

  • Loop entry / exit + collection binding. A new step kind (e.g. a loop / for-each node) that names the collection it iterates (e.g. the confirmed-inventory graphemes, or carved keys) and the sub-series (an ordered question id list / sub-flow) to run per element. Entry establishes the iteration set; exit is reached when the set is exhausted. This must be representable so the map projection (§C) can render it as a single “loop” node with the sub-series as a drill-down — not as N flattened copies.
  • Per-iteration state (the loop variable). A bounded, named iteration context — { index, item } — threaded into SurveyContext for the sub-series, so prompts can interpolate the current character/key and validate() can range-check against it. This is the structured replacement for the current cycle-is-an-error model: a loop is a bounded revisit driven by collection length, distinct from an unbounded goto back-edge, so advanceThrough’s visited-guard SurveyRunner.tsx:189 must learn to allow a loop re-entry while still rejecting genuine author-error cycles.
  • Per-iteration mutate() into an IR array. Each pass writes one element of an IR collection — e.g. per-character/per-key into stores[] or layers[].rows[].keys[]. The existing seam already supports this shape: pathAuthorizes authorizes array-index sub-paths under an array writes declaration (stores[] → stores[3].items) mutateApply.ts:90–96, and pb_standard_letters already rebuilds a stores[] array idempotently by stable name questions/b/pb_standard_letters.ts:105–125. The loop primitive would carry the iteration index/key into MutateContext so the per-pass patch targets the right array slot idempotently (re-running pass i replaces slot i, preserving M4). No mutate() signature change is forced; what’s missing is (a) the iteration context in MutateContext and (b) a deterministic per-element slot key.
  • Completeness-aware modelling. The loop must be expressible in the manifest/data graph as a bounded construct so C2 does not flag it as an illegal cycle — i.e. a loop node is a first-class graph element with an iteration cardinality, not a self-referential data edge. C5 (orphan inputs) and C7 (reachable) must understand the sub-series’ inputs are satisfied by the loop’s collection binding.

(3) Why this is first-class to the character-flow migration

The build-list / character-discovery experience is exactly the surface most likely to need “for each character, ask this sub-series” (e.g. placement, casing, or mark behaviour per grapheme). If the unification flattens it into a linear multi_select + linear questions, the per-element branching is lost — which is the opposite of the goal (“don’t dumb the flows down”). Therefore the loop primitive is part of the build-list promotion (§B / §F P3c), not a later nicety: the build-list module should be promoted as a loop-capable construct (or the loop primitive landed first, behind the flag), so the mature behaviour survives the move onto the modular spine.

Status: the schema does not support loops today and the runner/completeness layer treats cycles as errors. This is a genuine gap the unification must close before (or as part of) migrating the character-placement flows. It is flagged as Q5 / Risk R3 in §H, and it likely requires a survey/types.ts schema extension plus a MutateContext addition (iteration index/key) — possibly a contracts touch if the iteration key must round-trip in the IR.

E. Spine-vs-library split

Becomes first-class spine (the mature component flows)

  • Build-list — the default Phase B characters branch (modular question on mutate()).
  • Prefill confirm — modular display/read question at the head of the characters branch.
  • Track chooser — modular gate question driving the copy/adapt fork in data.
  • Carve / Mechanisms / Touch galleries — remain editor-steps (genuinely editor surfaces, not questions) but become first-class on mutate() and are surfaced by the manifest projection (§C). Touch is furthest along (#825).

Demoted to the vestigial library (inert-but-preserved)

  • The pb_* step-by-step battery (55 modules) — kept reachable via the discovery-method gate, but off the default spine; rendered as library-not-in-flow reserve nodes where not on the active branch buildStepGraph.ts:150–182.
  • The full non-identity Phase A battery (the 15 identity + 15 provenance_* questions) — the orphaned structure; demoted to library since the identity_lite head is the canonical experience.
No-delete guardrail (plan §3.8). Demotion is not deletion. Unreferenced modules stay registered in their sub-registries and on disk; they are simply absent from the active flow ordering, so they render as reserve/library nodes (the existing computeReserveNodes mechanism) and the runtime never reaches them. They remain test-covered and revivable by re-adding their id to a YAML branch. Article VII / the spec’s “no library/reserve deletion” rule is honoured.

F. Migration sequencing — small, independently-shippable, each landing green

Order chosen so the lowest-risk, highest-leverage change lands first and each step is behind the flag or read-only.

  1. Map-as-projection (read-only, no flag). no behaviour change Switch the dashboard’s flow section to render from buildManifestStepGraph() for the spine, with the per-phase modular graphs as registry-keyed drill-downs (§C). Purely additive to a developer-only view (gated by VITE_SHOW_FLOWMAP StudioShell.tsx:84); zero runtime/IR impact. Immediately makes the galleries visible and kills the dead stub legend. Ship first.
  2. Declare-only: galleries + build-list get real inputs/writes. declared, not executed Populate the currently-empty inputs: []/writes: [] on the carve/mechanism/touch editor-steps registerEditorSteps.ts and add the new build-list / track / prefill modules with declared paths. No mutate() executes yet (flag off). This lights up C5/C7 in the map and proves the data graph is consistent before wiring writes.
  3. Per-flow modularization behind the flag. VITE_KM_MUTATE_SEAM One flow at a time, each its own PR: (a) track chooser → modular gate; (b) prefill → modular read question; (c) build-list → modular mutate() writer (as a loop-capable construct — see §D2, landing or co-landing the loop primitive); (d) carve, (e) mechanisms routed through the reducer’s overlay-preserving setWorkingIR instead of direct store mutators. With the flag off, each PR is byte-identical to today (the reducer no-ops reducer.ts:205–217). Each lands green independently.
  4. Parity-prove and flip the default. flag on Once flag-on equals flag-off for every converted flow (§G), flip VITE_KM_MUTATE_SEAM on in dev/preview, soak, then default-on. Rollback remains “leave it off” (F2). Only after this do the direct store carve mutators become dead and can be retired; the pb_* battery and full Phase A are demoted to library (§E) under the no-delete guardrail.

G. Test strategy & parity proof (plan §7 two-tier)

Tier 1 — per-question unit tests (mirrored tree)

Every promoted module gets a colocated spec in the existing mirrored tree packages/studio/tests/survey/questions/<phase>/<id>.test.ts (the tree already exists: tests/survey/questions/{a,b,f}/). Per the mutate() obligations (contracts/mutate-seam.contract.md, lines 26–32): applying mutate() to a known IR fixture (1) changes exactly the declared writes paths, siblings byte-identical (M2/SC-002); (2) an out-of-writes patch fails fast, whole-patch rejected, IR unchanged (M3) — exercised against MutatePatchContainmentError mutateApply.ts:35–51; (3) idempotent (M4); (4) round-trips against reused IR fixtures. New gallery write surfaces add batch-patch fixtures; prefill is a read-only module so its test asserts declared inputs satisfiability, not a patch. A loop construct (§D2) additionally needs per-iteration tests: pass i writes only slot i, re-running pass i is idempotent (M4), and the empty-collection case is a clean no-op.

Tier 2 — mirrored / structural test tree

Manifest-shape guards already exist and must stay green: validateManifestShape() (M2 spine order, M3 locks, M4/M4b side-trails, M5 unique ids) StudioShell.tsx:253–303, and the completeness checks C1–C7 completeness.ts. Promoting questions must keep C3 (rejoin), C5 (orphan inputs), C7 (unreachable) passing — these are the structural proof that a promoted question is on a real, reachable branch with satisfied reads. A loop construct must keep C2 green (a bounded loop is not an illegal data cycle).

Parity proof — flag-on/flag-off equivalence

  • Baseline: with VITE_KM_MUTATE_SEAM off, produced IR + observable survey behaviour are byte-identical to P4b (F2/SC-008) — the reducer’s isMutateRequest guard returns before any write reducer.ts:205–206.
  • Equivalence test per converted flow (the “#832 50/50” parity style): run the same answer sequence through the flow with the flag off and with the flag on, and assert the resulting KeyboardIR is equal. Off path writes via the legacy route (store mutators / phaseResults); on path writes via mutate(). They must converge to the same IR — that is the proof the seam doesn’t change output, only the route. A converted flow does not flip its default until its 50/50 parity test is green. For the looping build-list, the parity sequence must include a multi-element collection so the per-iteration writes are exercised on both paths.
  • Boundary & round-trip: pnpm depcruise (dashboard→stores forbidden; the projection change must keep the dashboard store-free, props-only DashboardView.tsx:11–14, 301–309); contracts round-trip tests if any new IR field is added (§D / §D2).
  • Full green gate (mirrors plan T037): pnpm typecheck, studio + contracts vitest, pnpm depcruise, and flag-off output matching the recorded baseline — each migration PR must pass all of these.

Q5 — looping
Looping / iterative question-series (§D2). The schema supports no iteration today and the runner/completeness layer treats cycles as errors SurveyRunner.tsx:180–205 completeness.ts:182–185. The mature character-placement flows likely need “for each character, ask this sub-series.” Decision: do we land a first-class loop primitive (schema extension on FlowQuestion/FlowDef + an iteration index/key in MutateContext) before/as part of the build-list promotion, and does the iteration key need to round-trip in the IR (a possible contracts touch)? Without it, the build-list cannot be modularized without flattening away its looping behaviour.
Risk R1
Carve overlay semantics. The deletion overlay + undo stack is non-trivial; expressing removal as a declared-writes patch must preserve the overlay (the reducer already has an overlay-preserving setWorkingIR StudioShell.tsx:462–468, but the patch shape for “remove” vs “set” needs design). Highest-effort conversion; sequence it after the simpler writers.
Risk R2
Default-flip blast radius. Flipping VITE_KM_MUTATE_SEAM on changes the live write route for all converted surfaces at once. Mitigated by the per-flow 50/50 parity gate (§G) and the “leave it off” rollback, but the flip itself is the riskiest single moment — soak in preview first.
Risk R3
Loop primitive is a schema change with broad reach. Adding iteration touches the question schema, the SurveyRunner visited-guard, the map projection, and the completeness cycle check (C2 must distinguish a bounded loop from an illegal cycle). It is the most architecturally invasive piece of the unification — scope and sequence it deliberately (land the primitive behind the flag, with its own parity + completeness tests, before promoting the looping character flows).
