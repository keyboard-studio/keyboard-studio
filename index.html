Why it exists

The expertise gap in keyboard authoring

Language experts know their phonology, orthography, and character inventory. But shipping a keyboard to keymanapp/keyboards today means crossing a wall of mechanics.

Today's barriers

  • Learn .kmn rule syntax — stores, deadkeys, contexts, indexing.
  • Keep a half-dozen package files consistent by hand.
  • Satisfy ~200 PR-review criteria before a keyboard is accepted.
  • Compile and debug a toolchain that was never built for non-programmers.

What Studio removes

  • Answer questions about characters and behavior — never see syntax.
  • The studio generates and keeps in sync every package artifact.
  • A language-aware lint engine blocks invalid output before submit.
  • Delivery is a one-click .zip or draft PR via GitHub OAuth.
How it works

Survey → strategy → gallery → compile → deliver

Five stages take a language expert from a blank page to a validated, submittable keyboard.

01

05
🚀

Delivery

Core capabilities

What makes it work

🔤

Never write .kmn

The survey and gallery are the entire authoring surface. Generated KMN is an output, not an input.

🧩

Pattern schema as contract

Every interaction is a typed Pattern with named {{slotId}} placeholders — the Day-1 locked contract between engine and content teams.

🛡️

Three-layer lint

Layer A validity (14 compiler checks) + Layer B style + Layer C hygiene, language-aware, in two packages.

⚡

Single 300 ms cycle

TS-check and the WASM compiler oracle run as concurrent microtasks in one debounce — instant feedback.

💾

In-memory virtual FS

All authoring mirrors the keymanapp/keyboards layout in memory; nothing touches disk until output.

✅

criteria.md compliance

The ~200-item PR-review checklist surfaces in-studio, so submissions clear review the first time.

Strategy engine · spec §7

Twelve strategies, chosen for you

The survey computes seven discovery axes (character count, combining behavior, key availability, …) and runs a decision tree to recommend the right output method. Each strategy is a citable card with verbatim exemplars from real shipped keyboards.

From answer to rule

A user who says "I press a quote key, then a vowel, to add an accent" lands on S-02. The gallery surfaces the matching Pattern, the user fills the slots by tapping live keys, and the studio emits KMN like this — compiled and linted on every keystroke.

Real exemplar in the catalog: sil_euro_latin.kmn, whose 92 deadkey rules cover nearly every European Latin diacritic.

Scope · v1

What v1 covers

✓ In scope

  • Physical + touch keyboard layouts
  • Package generation — .kps / .kvks / .keyman-touch-layout
  • QWERTY/QWERTZ, AZERTY, and Non-Roman script groups
  • In-browser compile and three-layer lint

✕ Out of scope for v1

  • CJK and Ethiopic reorder patterns
  • LDML output
  • Predictive-text wordlists
  • Editing existing keyboards
